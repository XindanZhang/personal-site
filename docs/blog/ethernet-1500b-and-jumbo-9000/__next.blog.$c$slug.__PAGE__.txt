c:T996,<p>When people say <code>Ethernet 1500B</code>, they usually mean the normal Ethernet MTU or payload size, not the total frame size on the wire. <a href="https://datatracker.ietf.org/doc/html/rfc894">RFC 894</a> says the Ethernet data field for an IP datagram goes up to <code>1500</code> octets. In other words, <code>1500 bytes</code> is the familiar payload limit for ordinary Ethernet IP traffic.</p>
<p>That is also why the total frame is a little bigger than <code>1500 bytes</code>. Red Hat's performance guide explains that a standard Ethernet frame without a VLAN tag is <code>1518 bytes</code> in total, with <code>18 bytes</code> of Ethernet overhead and <code>1500 bytes</code> left for payload. So the common <code>1500</code> number is the data part people care about most, not the whole frame envelope.</p>
<p><code>Jumbo 9000</code> usually means configuring an MTU around <code>9000 bytes</code> instead of <code>1500 bytes</code>. The useful detail is that jumbo frames are non-standardized, so <code>9000</code> is a common operational choice rather than a single universal Ethernet rule. It is popular because larger frames reduce per-packet overhead and can lower CPU work when a host is moving large, contiguous streams such as backups, storage traffic, or lab data sets.</p>
<p>The catch is that jumbo frames only help when the whole path agrees. Red Hat's networking documentation is very direct here: all network devices on the transmission path need to support jumbo frames and use the same MTU. If one hop stays at <code>1500</code>, you can end up with fragmentation, drops, or confusing throughput regressions.</p>
<p>My short version is simple: <code>1500</code> is the safe default for general Ethernet networks, while <code>9000</code> is a deliberate tuning choice for controlled environments. If the traffic is large and the path is fully under your control, jumbo frames can be worth it. If not, <code>1500</code> is still the number to trust.</p>
