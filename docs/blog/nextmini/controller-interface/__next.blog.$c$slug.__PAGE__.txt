b:T1041,<h2 id="introduction-of-controller-interface">Introduction of controller interface</h2>
<p>The controller interface is a very important control-plane adapter for dataplane nodes. It bootstraps the nodes. The <code>controller_interface</code> is initialized in <code>Conductor::new()</code>. Then we build the <code>ControllerInterfaceHandle</code>.</p>
<h2 id="fn-connect">fn <code>connect()</code></h2>
<p>Once controller interface calls <code>connect()</code> to connect to the <code>controller</code> with <code>controller addr</code> via <code>WebSocket</code>, and if succeeds, dataplan node sends the <code>DataplaneToController::StartUp</code> message to <code>controller</code>, including the private/public addresses, network name, and requested <code>node_id</code> of each node. Then the dataplane node will wait for the controller's <code>startup</code> response containing the node’s assigned runtime configuration. The dataplane applies that config locally first, and only then starts the rest of the node runtime.</p>
<p>Thus in <code>connect()</code>, we have this kind of one-time <code>startup handshake</code> between a dataplane node and the controller. So it's literally the bootstrap exchange. It runs once per controller connection and is used to register the dataplane node and fetch <code>controller-assigned</code> config.</p>
<p>The one-time StartUp message is the exception. In <code>connect()</code>, it is written directly to ws_stream before the stream is split into sender and receiver halves.</p>
<h3 id="psuedocode-of-connect">psuedocode of <code>connect()</code></h3>
<pre><code class="language-rust">fn connect(config) -> (updated_config, processors, ws_stream):
    parse controller URL
    connect to controller websocket with retry loop and timeout
    send StartUp message // DataplaneToController::StartUp
    wait for one controller response
    if response is ControllerToDataplane::StartUp:
        update config from response
    create processor handle using updated config
    return updated_config, processors, ws_stream
</code></pre>
<h2 id="processor-started">Processor started</h2>
<p>Besides that, as you can see, the processor will be created after updating the <code>controller-assigned</code> configurations from the controller and, right before other handles which need to use <code>processor</code> to avoid the race condition.</p>
<p>Once <code>connect()</code> is done, <code>lossless_runtime</code> and <code>python_interface</code> will be created if needed. <code>processor.connect_lossless_handle()</code> is executed here to broadcast <code>ConnectLosslessHandle</code> to store <code>LosslessRuntimeHandle</code> to every processor for future use. We can find the <code>broadcast_receiver</code></p>
